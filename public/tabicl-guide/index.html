The Black Box

Before we open it up, here is the whole system in one diagram. You feed TabICLv2 a table (some training rows with labels, one test row without) and it hands you back a prediction. No training. No gradient descent. No fitting on your data. Just a forward pass.

Figure 1. TabICLv2 as a black box. Training rows go in, the test row's prediction comes out. No fine-tuning, no per-dataset training run. The same color scheme runs through every figure in this post: training rows test row (C_004) weight matrices predictions

The whole post is about what's inside that box and why it works. Let's open it.

Why Tabular Data Is Hard

Text has a fixed vocabulary. Images have a fixed pixel grid. A model trained on English can read any new English sentence. A model trained on ImageNet can process any new RGB image of the right shape. The input space is shared.

Tables have none of that. Every dataset has different columns, different scales, different types. A model trained on the Amex default table can't look at the iris flower dataset without retraining. There's no shared "tabular vocabulary." For the last twenty years the answer has been the same: train a fresh model from scratch for every new table. Gradient-boosted trees won that era by being very good at this from-scratch fitting.

Then TabPFN arrived. It showed you can pre-train a transformer on millions of synthetic tables, and what it learns is a general skill: read a tabular context, predict a held-out row. Once that skill is in the weights, you don't need to train per-dataset anymore. TabPFN, TabPFNv2, TabICL, and TabICLv2 all live in this paradigm.

The Foundation-Model Idea

Here's what changed in one diagram.

Figure 2. Classical ML fits a model to your data, so every new dataset means a new training run. TabICLv2 is pre-trained once, on a generative process that samples a fresh synthetic dataset every step. After that, any table you hand it is just context for a forward pass.

The training data here is not real tables. It's a generative process that samples a fresh synthetic table on every step. We'll come back to it in section 14. For now, the only thing that matters is that pre-training teaches one general skill: read a tabular context, predict the held-out row. That skill goes into the weights. Anything dataset-specific stays in the input.

Now we open the box.

Three Stages

TabICLv2 has three stages. Each does something different.

Stage 1 (TF_col) works one column at a time. For each column, all the rows' values for that column attend to each other. So all the balance_ratio values across rows talk to each other. Same for spend, days_delinquent, and the rest. The stage runs independently for each of the m columns.

Stage 2 (TF_row) flips it. Now we work one row at a time. For each row, the m features attend to each other. A [CLS] token gets prepended and pools the result into a single vector for that row.

Stage 3 (TF_icl) is in-context learning. Each row is now a single vector. The test row's vector queries the training rows' vectors, looks for ones that resemble it, and pulls their label signal back.

Figure 3. The three stages. Stage 1 looks down each column, so values in the same column from different rows compare against each other. Stage 2 looks across each row and pools the features into a single vector. Stage 3 runs in-context learning over those row vectors. The whole pipeline runs L times in stacked layers.

Before any stage can run, the table needs to be in a form a transformer can chew on. Numbers turn into vectors. The next two sections handle that. They're also where the first two TabICLv2-specific ideas live.

Turning a Row Into Vectors, Part 1: Feature Grouping

Throughout this post we'll use one running example. A small slice of the Amex default-prediction dataset, with five features per customer.

Figure 4. Our running example. The pattern is intentionally obvious: high balance ratio + low payment ratio + many days delinquent points to default. C_004 looks a lot like C_001 and C_003. So when the model attends across rows, it should pull most of its signal from those two, both of whom defaulted. In the real Amex dataset these features map to B_1, S_3, P_2, D_39, and R_1.

The first step is to turn each row into a sequence of vectors, one per feature. The naïve way: embed each column independently, each with its own linear layer. TabICL did this. It runs into a problem when two columns have similar distributions. Take balance ratio and risk score: both live in [0, 1], both lean high for risky customers. Independent embedding can produce nearly identical vectors for them. The model loses the ability to tell them apart. People call this representation collapse.

TabICLv2's fix is repeated feature grouping. Instead of embedding each column alone, each column gets bundled with two neighbors via circular shifts: column j goes with columns j+1 and j+3 (mod m). For our 5-column table, that's five overlapping triples.

Figure 5. Repeated feature grouping on C_001. Each of the five raw features generates one triple (itself plus its +1 and +3 neighbors, modulo 5). Every triple goes through the same linear layer to produce one d-dimensional embedding. Result: still five embeddings (the count is preserved) but each one now depends on three columns, breaking the symmetry that caused representation collapse in TabICL.

The shift pattern (0, 1, 3) isn't arbitrary. It's picked so that for any table with seven or more columns, no two features ever land in the same triple twice. The model sees each column in three different contexts. Never the same context twice.

Turning a Row Into Vectors, Part 2: Target-Aware Embedding

Now the second trick. The model needs to know which rows are training rows (with labels) and which is the test row (without). The earlier TabPFNv2 handled this by tacking the target on as an extra column. TabICLv2 does it more cleanly: it adds the target embedding directly into every feature embedding of every training row.

So for C_001 (label = default), an embedding for "default" gets added to all five of C_001's feature embeddings. Same for C_002 (no default) and C_003 (default). C_004 is the test row. Nothing gets added to it. From this point on, every layer can tell training rows from the test row just by checking whether the target signal is there.

Figure 6. Target-aware embedding. The model adds a label-specific vector to every feature embedding of every training row. The test row gets nothing added, and that absence is itself a signal. The paper's ablations identify this as one of three components (alongside the Muon optimizer and QASSMax) that each contribute roughly 100 Elo on validation datasets.

Stage 1: Looking Down a Column

Stage 1 runs one column at a time. Pick a column, say days_delinquent. We have one value of it for each row. Across our four rows that's 14, 0, 22, 18. Each value already became an embedding after grouping and target injection. Now those four embeddings attend to each other.

Why this is useful: a single value is hard to read on its own. Is days_delinquent = 18 high or low? Compared to the other rows in the table, it's roughly average. By letting cells in the same column see each other, the model learns each value's relative position in the column. That's something a fixed embedding can't give you, because what counts as "high" depends on the dataset.

The same thing then happens for every other column. balance_ratio across rows. spend across rows. And so on. Each column does its own pass, in parallel.

Figure 7. TF_col operating on the days_delinquent column. Each row's value for that column attends to every other row's value for the same column. After this stage, each cell encodes not just its own value but how it compares to the rest of the column. Technically TF_col is a Set Transformer, which uses learnable "inducing points" so the cost stays linear in n rather than quadratic.

Stage 2: Compressing Each Row

After stage 1, every cell is "context-aware." It knows where its column-mates sit in the distribution. Good. But for stage 3 to work, we need each row to be a single vector, not five. That's what stage 2 does.

For each row, we prepend a [CLS] token and run a transformer over the row's m+1 cells. Inside that transformer, all the features of the row attend to each other and to [CLS]. The output we keep is just [CLS]'s final representation. That's our row vector. The same trick BERT uses to summarize a sentence into a single embedding.

Figure 8. Row pooling via TF_row. A learned [CLS] token is prepended to the row's enriched feature embeddings. After a transformer pass, the [CLS] output is the row vector: one summary vector per row. This is what enters stage 3.

The Sequence of Row Vectors

After stages 1 and 2 we have four row vectors. One per row. The first three carry their target signal from way back in the embedding step. The fourth doesn't. That one's the test row.

Figure 9. The four row vectors after stages 1 and 2. Three training rows (teal) carry their label signal through the target-aware embedding. The test row (dark cyan) does not. This is the sequence of tokens that enters stage 3.

Stage 3: Rows Talking To Each Other

This is where in-context learning happens. C_004's row vector needs to find which training rows look like it. Then it needs to pull their label signal back.

Transformers do this with three projections per token. Query, key, value. Three learned weight matrices. Each row vector goes through all three and produces three new vectors.

Figure 10. Each row vector gets projected three ways through three shared weight matrices. The output is a query, a key, and a value for every row. C_004's query is what we'll match against everyone else's keys.

First, take the dot product between C_004's query and every training row's key. One similarity score per row.

Figure 11. Dot-product scores. C_004 looks like C_001 and C_003 (both defaulted), and not much like C_002 (no default). These numbers are illustrative, picked to match the obvious pattern in our toy table.

Second, divide by √d_k. d_k is the head dimension. Without this division, dot products grow with the dimension and softmax becomes too sharp. Then run softmax. The scores turn into a probability distribution that sums to 1.

Figure 12. Attention weights after softmax(scores / √d_k). Most of the attention lands on C_001 and C_003, the two defaulters. The combined weight on default-class rows is …. About 94% of C_004's attention is going to customers who defaulted. That's what dominates the next step.

Third, take a weighted sum of the value vectors using those attention weights. The result is a new vector for C_004, enriched by the rows it most resembles. Those rows carry the "default" signal we injected all the way back in stage 1. So that signal flows through and dominates C_004's enriched representation.

What just happened

The training labels were never something the model had to predict in this step. They were baked into the row vectors as input features, way back at target-aware embedding. By the time we reach stage 3, "find similar rows" and "copy their label" have collapsed into the same operation. Similar rows literally have the label signal sitting in their value vectors. Pull from them, get the label.

Why Q, K, V Are Three Separate Things

It looks like overkill. Three projections of the same vector. Here's the library analogy that makes it click.

You walk into a library looking for something. The query is what you want ("books about default risk"). The key is what each book advertises on its spine ("Credit Risk Modeling"). The value is what's actually inside the book: the chapters, examples, equations.

You match your query against the keys to figure out which books to pull off the shelf. Then you read the values to learn something. Keys advertise. Values deliver. Separating them lets the model learn one thing for "what should match" and another for "what to extract once matched." For tabular ICL that's exactly what we want: match rows on their features, but extract their label-flavored representation.

Masking

Masking does two jobs in TabICLv2. They're easy to confuse.

Label masking. The test row's label is hidden from the input. C_004 enters the pipeline with no target embedding added. Without this, the model would learn to copy the label directly and never need to attend to anything. The prediction task only exists because we hide the answer.

Attention masking. Inside TF_icl, no row should attend to the test row's position. The test row carries no label information, so attending to it would just dilute everyone else's signal with noise. The fix is the standard transformer trick: set those positions in the attention matrix to −∞ before the softmax.

Figure 13. The two jobs of masking. Left: C_004's label is removed from the input, which is what creates the prediction task in the first place. Right: in TF_icl, the attention matrix has C_004's position blocked with −∞ scores, so softmax sends those weights to exactly zero. No row, including C_004 itself, attends to a label that is not there.

The −∞ trick works because softmax exponentiates first. exp(−∞) = 0, so the weight on that path is exactly zero. No need to special-case anything. Just inject −∞ before the softmax, the rest of the math takes care of itself.

Attention Fading and QASSMax

Softmax has a problem at scale. Our four-row example hides it, but real tables have thousands of rows. Maybe a million.

The denominator of softmax sums over every row. The more rows you add, the bigger that sum gets. Even if C_004 has a strong match with one anchor row, that match gets divided by a denominator that keeps growing. The attention weight on the anchor row drops toward zero as n grows, even though nothing about the match itself changed. People call this attention fading.

It bites foundation models specifically. TabICL was pre-trained on contexts up to 60K rows. Push it past that and the attention distribution flattens. Every row gets roughly the same weight. The model can't focus.

TabICLv2's fix is QASSMax, short for Query-Aware Scalable Softmax. The idea is small. Scale the queries by something that grows with log(n) before you compute softmax. As n grows, queries get scaled up, dot products get bigger, and softmax stays sharp. A recent result shows log(n) is the right rate.

Below: the left chart is plain softmax, the right is QASSMax. Both see one anchor row that's a real match and (n−1) noise rows that aren't. The slider changes n.

The real QASSMax is a bit fancier. The scaling isn't just log(n) · s. It's a small two-layer MLP that takes log(n) as input, plus a query-dependent gate. Same idea, more expressive.

Where the Weights Came From

All those weights (W_Q, W_K, W_V, the linear layers, the QASSMax MLPs) have to come from somewhere. This is the training loop.

Figure 15. The training loop. About 24.5 H100-days total, ~550K iterations across three stages, batch size 64. Each iteration samples a fresh synthetic table.

The training data is not a fixed dataset. It's a generative process. Each step samples a fresh synthetic table from a procedural prior. The recipe: sample a random DAG, place a random function at each node (MLP, tree ensemble, Gaussian process, polynomial, plateau function, or product of these), feed random vectors in at the roots, propagate through, extract some columns as features and one as the target. Throw away datasets where target is independent of features. Repeat about 35 million times.

Now the punchline. Every iteration uses a different dataset. The optimizer can't lower loss by memorizing one table, because by next step that table is gone forever. The only thing that consistently lowers loss across millions of distinct tables is the general skill of reading a tabular context and predicting the held-out row. That skill is what gets compressed into the weights.

In meta-learning terms: the prior is the inner-loop task distribution, and the outer loop is gradient descent on the model's ability to handle that whole distribution at once. The model isn't learning a task. It's learning how to learn tasks like these.

The loss is the expected negative log-likelihood of the test row's true label, averaged over all sampled datasets. Every gradient step nudges W_Q and W_K. The push is in the same direction every time: make rows that share a label produce higher dot products than rows that don't. That's the only thing that consistently helps, no matter which dataset comes up next.

Stacking Layers

One pass of TF_col → TF_row → TF_icl isn't enough. TabICLv2 stacks the whole thing into L layers. Early layers do rough work: simple feature interactions, coarse row-to-row matches. Deeper layers refine these into sharper, more abstract patterns.

Figure 16. The full stack. Each layer takes the previous layer's row vectors, runs them through the same three-stage architecture, and produces refined row vectors for the next layer. The W_Q, W_K, W_V matrices are different per layer; every layer learns its own attention patterns.

If you remember one thing

TabICLv2 takes a table, reads it once, predicts. The "training data" lives in the context window, not the weights. The weights store the general skill of reading tabular context, a skill it learned from tens of millions of synthetic tables, none of which it ever saw twice.

What We Glossed Over

A few things were simplified to keep the diagrams readable. None of them change the picture above. They're the kind of details that turn into footnotes once you have the main idea.

Every QKV walkthrough used a single attention head. The real model uses 8 heads in TF_col and TF_icl. Each head is a separate "match on this kind of similarity, extract this kind of value" pattern, learned independently. Their outputs are concatenated. Same intuition, more capacity.

When we said TF_col is a Set Transformer, that's more than a label. Set Transformers use inducing points: a small fixed-size set of learnable queries that all input tokens attend to first, with outputs read back from those inducing points. This keeps attention linear in the number of features rather than quadratic. That's how TabICLv2 stays fast on tables with hundreds of columns.

We described classification throughout. Regression works differently. TabICLv2 swaps the classification head for one that predicts 999 quantiles, trained with pinball loss. The rest of the architecture stays identical. Most tabular foundation models either bin the target into a categorical or predict a single point. The quantile head gives you a full predictive distribution for free.

We also assumed at most 10 classes. TabICLv2 was pre-trained with up to 10, and beyond that it uses a trick called mixed-radix ensembling. Decompose the class label into multiple "digits" in different bases (each ≤10). Run the model once per digit. Recombine. It's its own small idea, worth a separate post.

The pretraining setup is also worth a look. Three stages, with dataset sizes climbing from 1,024 rows in stage 1 to up to 60K in stage 3. The optimizer is Muon, not AdamW, which lets the model use higher learning rates (8e-4 in stage 1 vs. 1e-4 for AdamW in TabICL). There's cautious weight decay too. It only fires when the gradient and parameter agree in sign, so beneficial gradient directions aren't interfered with. Total cost: about 24.5 H100-days. Cheap by foundation-model standards.

The synthetic prior gets its own appendix in the paper. Eight types of random functions, multiple random graph topologies, dataset-level filtering. The prior turns out to be the single highest-impact ablation. Training the v2 architecture on the v1 prior fails outright. Most of the engineering effort behind TabICLv2 lives in this generative process.

For all of these, read the paper. Architecture is in §3, pretraining in §4, the synthetic data engine in §5, ablations in §7, and the quantile-distribution math in Appendix I.
